## 1. Spatial Metaphors for LLM Memory: MemPalace
- **来源**：arXiv cs.AI
- **链接**：https://arxiv.org/abs/2604.21284
- **一句话解释**：利用古老的“记忆宫殿”法，将 LLM 的长期记忆组织成虚拟的空间结构。
- **它为什么重要**：它把枯燥的向量数据库（Vector DB）变成了可感知的、具备空间隐喻的建筑。对于设计背景的你，这是将 RAG（检索增强生成）从“黑盒搜索”转向“空间设计”的信号。
- **技术趋势标签**：#HCI #SpatialComputing #LLM-Memory
- **可复用模式**：将非结构化数据映射到 3D/2D 坐标系中，通过“游走”进行检索而非关键词匹配。
- **疯狂发明家灵感**：
  1. **一个小实验**：写个脚本，把你的所有 Obsidian 笔记按主题生成一个 Minecraft 地图，每个房间放一个包含笔记摘要的告示牌。
  2. **一个中型项目**：开发一个 AR 浏览器插件，根据你浏览网页的类型，在你的房间角落“放置”虚拟书架，下次想找该网页时，直接看向那个角落。
  3. **一个艺术/游戏/叙事方向**：一个 NPC，它的记忆受限于它在游戏世界中的“物理家园”，如果它的房子被烧毁，它会丢失相关的逻辑记忆。
- **对我项目的启发**：你的 Agent 记忆不应该是 JSON，而应该是一个“景观”。
- **今天是否值得深入研究**：是
- **建议行动**：做 demo（试着用 A-Frame 或 Three.js 可视化你的 Embedding 空间）

## 2. Co-Evolving LLM Decision and Skill Bank Agents
- **来源**：arXiv cs.AI
- **链接**：https://arxiv.org/abs/2604.20987
- **一句话解释**：让 Agent 在长程任务（如游戏）中不仅学会做决策，还能自己提炼并“进化”出一个可重复使用的技能库（Skill Bank）。
- **它为什么重要**：解决了 Agent “每次都从零开始想”的低效。这正是 Minecraft 自动化或复杂系统生成的关键：从“动作”进化到“函数库”。
- **技术趋势标签**：#Game-AI #Skill-Learning #Agent-Workflow
- **可复用模式**：动作 -> 评估 -> 抽象为技能 -> 存储到 Skill Bank -> 再次调用。
- **疯狂发明家灵感**：
  1. **一个小实验**：让 LLM 写 Python 绘图代码，每成功画出一个形状，就自动将其封装成一个库函数，下次直接调用。
  2. **一个中型项目**：一个“会学习的 Minecraft 建筑师”，你给它看一张图，它尝试建造后，把“如何盖房顶”总结成一段代码存入自己的技能书。
  3. **一个艺术/游戏/叙事方向**：一个会随着玩家交互而“学习技能”的宠物，它学到的技能会通过 LLM 生成新的动画逻辑。
- **对我项目的启发**：你的 Spec-driven development 可以增加一个“经验回馈环”，让 Spec 自动进化。
- **今天是否值得深入研究**：是
- **建议行动**：试用（参考 Voyager 项目的 Skill Bank 实现思路）

## 3. WorkflowGen: Trajectory Experience Driven Generation
- **来源**：arXiv cs.LG
- **链接**：https://arxiv.org/abs/2604.19756
- **一句话解释**：通过分析 Agent 过去的执行轨迹（Trajectory），自动生成稳定的、可复用的自动化工作流。
- **它为什么重要**：它把“试错”变成了“代码”。如果你在做自动化系统，这能让你从“手写 Prompt 编排”解脱出来，进入“让 Agent 观察我操作并生成 Spec”的阶段。
- **技术趋势标签**：#Automation #Spec-Driven #Workflow-Optimization
- **可复用模式**：轨迹挖掘 -> 模式提取 -> 确定性工作流生成。
- **疯狂发明家灵感**：
  1. **一个小实验**：录制一段你整理文件的操作日志，让 LLM 将其转化为一个可运行的 Python 脚本。
  2. **一个中型项目**：一个“设计助手”，它观察你在 Figma 里的重复操作，自动弹出一个弹窗问：“要我帮你把这个工作流变成一个插件吗？”
  3. **一个艺术/游戏/叙事方向**：一个观察玩家游戏习惯并将其转化为“关卡生成规则”的元游戏。
- **对我项目的启发**：Drift 系统的核心可以是“从混乱的交互轨迹中沉淀出秩序（Workflow）”。
- **今天是否值得深入研究**：是
- **建议行动**：做 demo

## 4. Symbolic Grounding in Abstract Visual Reasoning (Bongard-LOGO)
- **来源**：arXiv cs.AI
- **链接**：https://arxiv.org/abs/2604.21346
- **一句话解释**：研究 VLM（视觉语言模型）如何通过“生成程序”来理解抽象的视觉逻辑问题（如 Bongard 问题）。
- **它为什么重要**：它证明了“代码是理解视觉逻辑的最好语言”。这对于做生成式艺术或 Minecraft 世界生成的你来说，是极强的理论支撑。
- **技术趋势标签**：#Creative-Coding #Generative-System #Symbolic-AI
- **可复用模式**：视觉输入 -> 猜测生成规则（代码）-> 验证代码 -> 理解逻辑。
- **疯狂发明家灵感**：
  1. **一个小实验**：给 LLM 看一张简单的几何构图，让它写出生成这张图的 p5.js 代码。
  2. **一个中型项目**：一个“逆向生成器”，给它一张 Minecraft 建筑截图，它能反推出生成该建筑的规则脚本。
  3. **一个艺术/游戏/叙事方向**：一个基于“视觉谜题”的解谜游戏，玩家必须通过编写简单的程序逻辑来改变世界。
- **对我项目的启发**：在你的生成式系统里，不要直接让 LLM 产出像素/方块，让它产出“生成逻辑的代码”。
- **今天是否值得深入研究**：否（除非你现在要做视觉推理底层）
- **建议行动**：收藏

## 5. Do MLLMs Understand Pointing? (Egocentric Vision)
- **来源**：arXiv cs.HC
- **链接**：https://arxiv.org/abs/2604.21461
- **一句话解释**：测试多模态大模型是否真的理解人类“指指点点”（Pointing Gestures）的含义。
- **它为什么重要**：这是 HCI 的核心。如果 Agent 要在现实世界或 VR 环境中辅助你，它必须理解你的手指指向哪里。
- **技术趋势标签**：#HCI #Multimodal #AR/VR
- **可复用模式**：空间坐标 + 视觉参考物 + 语言指令的联合推断。
- **疯狂发明家灵感**：
  1. **一个小实验**：用手机摄像头对准桌面，指着某个物体说“它是什么”，看 GPT-4o 的识别率，并尝试用代码标出你的指尖位置。
  2. **一个中型项目**：一个“AR 魔法指挥棒”，通过手指指向，实时控制智能家居或屏幕上的元素。
  3. **一个艺术/游戏/叙事方向**：一个全息 NPC，它会盯着你的手看，如果你指着它，它会害羞或生气。
- **对我项目的启发**：Agent 的交互不应局限于对话框，要考虑“空间语义”。
- **今天是否值得深入研究**：否
- **建议行动**：收藏

## 今日模式总结
1. **经验代码化 (Experience-to-Code)**：不再是手写代码，而是让 Agent 观察操作（轨迹）或观察结果（视觉），自动反推并生成可执行的代码/技能。
2. **空间隐喻 (Spatial Metaphor)**：LLM 的数据管理正从“列表式”转向“建筑式”，记忆宫殿法正在技术化。

## 今日可做小实验：【Memory-Palace-CLI】
**2 小时挑战：给你的项目文件夹盖一座“记忆宫殿”**

- **项目名**：Struct-to-Palace
- **输入**：一个包含多个 .py 或 .md 文件的项目文件夹。
- **输出**：一个 HTML/A-Frame 页面，将文件转化为 3D 空间中的“建筑”。
- **技术栈**：Python, OpenAI API (用于生成摘要), A-Frame (超简单的 WebVR 框架)。
- **第一步怎么做**：
  1. 写一个 Python 脚本遍历文件夹，读取文件名和内容。
  2. 调用 LLM，为每个文件生成一个“视觉隐喻”（例如：这个文件是“核心逻辑”，对应“地基”；那个文件是“配置”，对应“窗户”）。
  3. 生成一个简单的 A-Frame HTML 文件，把这些文件渲染成不同颜色和高度的方块（Box），点击方块显示文件摘要。
