## 1. [MemPalace：LLM 记忆的“记忆宫殿”空间隐喻]
- **来源：** arXiv cs.AI (2604.21284)
- **链接：** https://arxiv.org/abs/2604.21284
- **一句话解释：** 借用古罗马的“记忆宫殿”法（Method of Loci），将 LLM 的长期记忆组织成虚拟的空间地图，实现极高的检索召回率。
- **它为什么重要：** 它打破了传统 RAG 仅仅是“向量搜索”的枯燥逻辑，将数据结构**空间化、视觉化**。
- **技术趋势标签：** #Spatial_Computing #Memory_Architecture #HCI
- **可复用模式：** 将非结构化数据映射到 3D 坐标系，利用空间邻近性代替语义邻近性进行检索。
- **疯狂发明家灵感：**
  1. **一个小实验：** 用 Three.js 写一个简单的 3D 房间，把你的笔记挂在墙上，让 LLM 扮演“导游”在这个房间里边走边回答问题。
  2. **一个中型项目：** **"Infinite Minecraft Library"**：一个 Minecraft 插件，自动根据你的文档生成无限延展的图书馆建筑，书架的物理位置就是 LLM 的记忆索引。
  3. **一个艺术/游戏/叙事方向：** 创造一个“失忆症 AI 侦探”，玩家必须在物理空间中移动物品，才能帮 AI 找回特定的记忆碎片。
- **对我项目的启发：** 非常适合结合你关注的“世界生成”和“LLM 工具链”，让 Agent 的思考过程变得可见且可交互。
- **今天是否值得深入研究：** **是**
- **建议行动：** **做demo** (尝试把向量数据库的 Top-K 结果映射到 3D 坐标)。

## 2. [Co-Evolving LLM Agents: 游戏长程任务的技能银行]
- **来源：** arXiv cs.AI (2604.20987)
- **链接：** https://arxiv.org/abs/2604.20987
- **一句话解释：** 让 Agent 在游戏环境中一边决策一边进化出“技能库”（Skill Bank），通过不断复用和组合旧技能来解决超长链路的复杂问题。
- **它为什么重要：** 解决了 Agent“转头就忘”和“无法积累经验”的痛点，是通往 Minecraft 级复杂自动化系统的必经之路。
- **技术趋势标签：** #Agent_Workflow #Game_AI #Skill_Acquisition
- **可复用模式：** **“决策者-技能库”双回路架构**：决策层负责拆解，技能库负责执行微小动作原子。
- **疯狂发明家灵感：**
  1. **一个小实验：** 写一个 Python 脚本，让 LLM 玩一个极简的文字冒险游戏，并要求它每过一关总结一个“技能宏”存入 JSON。
  2. **一个中型项目：** **"Spec-Driven Architect"**：一个能自动编写并调试 Python 脚本的 Agent，它会把写好的工具函数（如爬虫、绘图）自动分类进自己的“工具箱”。
  3. **一个艺术/游戏/叙事方向：** 一个会“学习”玩家操作习惯的数字宠物，通过观察玩家的 Creative Coding 代码，进化出自己的绘画风格。
- **对我项目的启发：** 强化了你对 Agent workflow 的理解，特别是如何让 Agent 随着使用时间的增加而变得更“聪明”。
- **今天是否值得深入研究：** **是**
- **建议行动：** **收藏** (关注其 Skill Bank 的数据结构定义)。

## 4. [WorkflowGen: 基于轨迹经验的自适应工作流生成]
- **来源：** arXiv cs.LG (2604.19756)
- **链接：** https://arxiv.org/abs/2604.19756
- **一句话解释：** 不再每次从头生成工作流，而是通过分析过去的“成功执行轨迹”来快速拼装和调整新的自动化流程。
- **它为什么重要：** 极大地降低了 Agent 的推理开销，让自动化系统从“蛮力思考”转向“模式识别”。
- **技术趋势标签：** #Automation #Workflow_Orchestration #Efficiency
- **可复用模式：** **“轨迹缓存” (Trajectory Cache)**：存储成功的 Prompt 链条，作为未来任务的模板。
- **疯狂发明家灵感：**
  1. **一个小实验：** 记录你手动操作 AI 工具的 5 次记录（输入-输出对），喂给 Claude，让它为你生成一个专属的“工作流宏”。
  2. **一个中型项目：** **"Drift Auto-Pilot"**：为你自己的 Drift 系统增加一个“记忆层”，当它识别到当前任务类似之前的某个任务时，直接跳过推理步骤。
  3. **一个艺术/游戏/叙事方向：** 动态生成的关卡设计器：根据玩家过去解谜的路径（轨迹），自动生成难度曲线平滑的新关卡。
- **对我项目的启发：** 直接对应你的“自动化”和“LLM 工具链”兴趣，是优化 Agent 运行效率的关键。

## 5. [Symbolic Grounding: 抽象视觉推理中的符号化瓶颈]
- **来源：** arXiv cs.AI (2604.21346)
- **链接：** https://arxiv.org/abs/2604.21346
- **一句话解释：** 研究发现 VLM 在处理抽象视觉逻辑（如 Bongard 问题）时，瓶颈不在于推理，而在于无法将视觉特征转化为“生成程序”。
- **它为什么重要：** 证明了“代码作为中介”在生成艺术和复杂视觉任务中的核心地位。
- **技术趋势标签：** #Creative_Coding #Generative_Systems #Symbolic_AI
- **可复用模式：** **“视觉 -> 程序符号 -> 逻辑推理”** 的三段式路径。
- **疯狂发明家灵感：**
  1. **一个小实验：** 给 AI 看一张极简主义绘画（如蒙德里安），让它写一段 p5.js 代码复刻出来，观察它丢失了哪些“逻辑”。
  2. **一个中型项目：** **"Spec-to-P5" 编译器**：一个专门将模糊的视觉描述（如“要一种流动的、带有秩序感的混乱”）转化为严谨的生成艺术代码的 Agent。
  3. **一个艺术/游戏/叙事方向：** 视觉谜题游戏：玩家必须编写简短的代码来补全缺失的视觉逻辑图形才能过关。
- **对我项目的启发：** 强化了你作为 CS+Art 学生的直觉——代码是连接视觉感知与逻辑推理的唯一桥梁。
- **今天是否值得深入研究：** **是**
- **建议行动：** **做demo** (测试 LLM 的抽象视觉转代码能力)。

## 今日可做小实验：Spatial Prompt Palace (2小时)
*这是一个验证“空间位置是否能帮助 LLM 更好地管理上下文”的极简实验。*

- **项目名：** PromptRoom-01
- **输入：** 10 段互不相关的技术文档片段。
- **输出：** 一个 HTML 页面，包含一个 A-Frame (WebVR) 场景。
- **技术栈：** Python, OpenAI API, A-Frame (简单的 HTML 标签即可)。
- **第一步怎么做：**
  1. 使用 Python 将 10 段文档分别喂给 LLM，让它为每段文档分配一个 (x, y, z) 坐标（理由是：相关的文档坐标应接近）。
  2. 生成一个 HTML 文件，使用 `a-text` 标签将文档内容放在对应的坐标点。
  3. 问 LLM 一个跨文档的问题，但在 Prompt 中加入空间提示：“你现在站在 (0,0,0)，左边墙上是 A 文档，右边远处是 B 文档...”，看它的回答准确率是否比直接喂 10 段文字更高。
