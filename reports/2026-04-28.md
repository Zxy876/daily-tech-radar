## 1. Memanto: 信息论驱动的长程 Agent 语义存储
- 来源：arXiv cs.AI
- 链接：https://arxiv.org/abs/2604.22085
- 一句话解释：通过信息论方法优化 Agent 的持久化内存，解决 Agent 在长期任务中“记不住”或“记太杂”的架构瓶颈。
- 它为什么重要：目前的 RAG 往往是简单的向量检索，而 Memanto 试图建立一种“带类型的语义内存”，这对构建具有长期性格和记忆的 NPC 或自动化开发助手至关重要。
- 技术趋势标签：#Long-Horizon-Agents #Memory-Architecture #Information-Theory
- 可复用模式：**“选择性遗忘与类型化压缩”** —— 不是存下所有对话，而是根据信息熵提取关键状态变化。
- 疯狂发明家灵感：
  1. **一个小实验**：写一个脚本，将你过去一年的日记通过“信息熵”过滤，只生成 100 个反映你性格转变的“核心语义锚点”。
  2. **一个中型项目**：一个 Minecraft 长期生存 Agent，它不记录坐标，而是记录“情感地图”（哪里危险、哪里温馨），并基于此产生自主行动意图。
  3. **一个艺术/游戏/叙事方向**：一个“会失智的数字生命”，随着运行时间增加，它必须在存储新记忆和保留旧记忆之间做信息论上的权衡，导致其行为产生不可预见的漂移。
- 对我项目的启发：在做 Agent workflow 时，尝试在 Prompt 之外引入一个动态更新的 `state.json`，作为其“类型化内存”。
- 今天是否值得深入研究：是
- 建议行动：做demo

## 2. ruflo: Claude 驱动的多 Agent 编排平台
- 来源：GitHub (ruvnet/ruflo)
- 链接：https://github.com/ruvnet/ruflo
- 一句话解释：一个针对 Claude 优化的分布式多 Agent 群体智能编排平台，支持原生 Claude Code 集成。
- 它为什么重要：它把“群体智能（Swarm Intelligence）”工程化了。对于关注 spec-driven development 的你来说，这是将“需求文档”自动拆解给一群 Agent 执行的现成脚手架。
- 技术趋势标签：#Agent-Orchestration #Claude #Swarm-Intelligence
- 可复用模式：**“分布式集群任务分发”** —— 将复杂任务拆解为互不干扰的 Agent 节点。
- 疯狂发明家灵感：
  1. **一个小实验**：利用它的 Swarm 模式，让三个 Agent 分别扮演“杠精”、“吹鼓手”和“理中客”，针对你的一个发明点子进行辩论。
  2. **一个中型项目**：一个“全自动游戏设计工作室”，一个 Agent 写数值，一个写剧情，一个写 p5.js 代码，ruflo 负责协调。
  3. **一个艺术/游戏/叙事方向**：一个基于 Swarm 的实时生成叙事系统，每个 Agent 控制一个 NPC，它们在同一个服务器里“自主生活”并产生冲突。
- 对我项目的启发：直接观察它的任务分发逻辑，看它如何处理 Agent 之间的通信死锁。
- 今天是否值得深入研究：是
- 建议行动：试用

## 3. Self-Correction Cybernetics: 当 LLM 自我修正失效时
- 来源：arXiv cs.AI (Candidate 9)
- 链接：https://arxiv.org/abs/2604.22273
- 一句话解释：用控制论（Cybernetics）的反馈环模型来诊断：LLM 的“自我修正”什么时候在帮倒忙，并提出了“验证优先”的干预机制。
- 它为什么重要：你关注 spec-driven，而“修正”是其中最坑的一环。这篇文章提供了数学框架来判断：你的 Agent 是在变好，还是在陷入循环。
- 技术趋势标签：#Self-Correction #Cybernetics #Control-Theory
- 可复用模式：**“验证器-执行器双闭环”** —— 修正动作必须由一个独立的、逻辑更严密的“控制反馈器”来决定是否执行。
- 疯狂发明家灵感：
  1. **一个小实验**：写一个 Python 装饰器，监控 LLM 的连续三次输出，如果相似度过高或逻辑得分下降，强行中断并引入“随机噪声”扰动。
  2. **一个中型项目**：一个“不倒翁”代码生成器，它在生成代码时不断自我攻击测试，只有通过所有边缘案例测试的“修正”才会被保留。
  3. **一个艺术/游戏/叙事方向**：一个基于控制论的叙事 Agent，它不断修正自己的故事走向以维持读者的“惊讶度”在一个特定区间。
- 对我项目的启发：在你的自动化系统中，加入一个基于 Markov 模型的“健康度检查点”，防止 Agent 陷入死循环。
- 今天是否值得深入研究：否（看结论即可）
- 建议行动：收藏

## 4. Text-to-Model (TTM): 从文本到过程化技能模型
- 来源：arXiv cs.HC (Candidate 19)
- 链接：https://arxiv.org/abs/2604.17624
- 一句话解释：一种将教学材料自动转化为“任务-方法-知识（TMK）”结构化模型的 LLM 方法。
- 它为什么重要：这正是 Minecraft 世界生成或复杂游戏系统需要的——将模糊的指令转化为可执行的“过程化技能树”。
- 技术趋势标签：#Procedural-Skills #Knowledge-Representation #TMK-Model
- 可复用模式：**“语义解构为状态机”** —— 将自然语言动作拆解为：前提条件、执行步骤、预期结果。
- 疯狂发明家灵感：
  1. **一个小实验**：喂给 LLM 一段折纸教程，让它输出一个标准的 JSON 状态机，描述每一步的手部动作和纸张状态。
  2. **一个中型项目**：一个“Minecraft 建筑大师”Agent，它能阅读网上的建筑攻略，自动将其转化为一套通用的“建筑技能库”。
  3. **一个艺术/游戏/叙事方向**：一个“仪式生成器”，用户输入一种情绪（如“悲伤”），系统基于 TMK 生成一套怪诞但逻辑自洽的艺术仪式步骤。
- 对我项目的启发：这是实现“从 Spec 到实现”的中间件。不要让 Agent 直接写代码，先让它生成 TMK 模型。
- 今天是否值得深入研究：是
- 建议行动：做demo

## 今日模式总结
1. **控制论反馈 (Cybernetic Feedback)**：不再盲目相信 LLM 的自我修正，而是引入外部观察者和反馈环来控制生成质量。
2. **层次化技能建模 (Hierarchical TMK)**：从“直接生成内容”转向“先生成结构化的过程模型，再执行”，这是解决复杂系统生成的关键。

## 今日可做小实验：2 小时挑战
### 项目名：**"仪式感"代码重构器 (The Ritual Refactorer)**
- **输入**：一段你写得很烂的 Python 代码 + 一个“仪式感”关键词（如：哥特式、极简主义、混乱邪恶）。
- **输出**：一段逻辑等价但风格被彻底“仪式化”的代码，并附带一份 TMK 格式的“重构逻辑说明”。
- **技术栈**：Python, OpenAI SDK (or Claude API), Pydantic (用于定义 TMK 结构)。
- **第一步怎么做**：
  1. 定义一个 Pydantic 类 `RefactorStep`，包含 `pre_condition`, `action`, `semantic_meaning`。
  2. 写一个 Prompt：“你是一个代码祭司。请将以下代码重构为[风格]，并必须按照 TMK 结构记录你的每一步‘仪式’逻辑。”
  3. 运行代码，观察 LLM 如何将“重构”这种工程行为解释为“艺术仪式”。
